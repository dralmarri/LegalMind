# RETRIEVAL_ENGINE — محرك الاسترجاع

> **الحالة: ❌ غير موجود.** لا يوجد **أي كود يقرأ من Qdrant** في المشروع كله. Qdrant اليوم مخزن للكتابة فقط.
> `knowledge-system/RETRIEVAL_ENGINE.md` مواصفة نظرية بلا تنفيذ.

## 1. العائق الأول — التضمين ليس دلاليًا 🔴

`engine/legalmind_engine.py::hash_embedding()`:

```python
digest = hashlib.sha256(token.encode()).digest()
index  = int.from_bytes(digest[:4], "big") % 384   # hashing trick
sign   = 1.0 if digest[4] % 2 == 0 else -1.0
vector[index] += sign
```

هذا **ليس نموذج تضمين**، بل *hashing trick* على حقيبة كلمات. كل كلمة تُسقَط في خانة عشوائية ثابتة.

**الأثر القانوني:**

| الاستعلام | الوثيقة | نتيجة hash_embedding |
|---|---|---|
| «الحضانة» | «حضانة الصغير» | متجهان **شبه متعامدين** |
| «رعاية المحضون» | «الحضانة» | **لا تشابه** |
| «متاع الزوجية» | «المصوغات ومنقولات البيت» | **لا تشابه** |

المرادفات القانونية العربية — وهي **جوهر البحث القانوني** — لن تُسترجع أبدًا. المحامي الذي يسأل عن «مسكن الحاضنة» لن يجد `JUR-PS-JURISDICTION-00021` رغم أنه المبدأ الحاكم بالضبط.

الأسوأ: النظام سيعيد نتائج **بثقة**، وهي فارغة أو خاطئة.

**القرار:** `hash_embedding` **يجب أن تُحذف**، لا أن تُحسَّن. لا يُبنى استرجاع فوقها.

## 2. البديل — نموذج تضمين عربي حقيقي

الاشتراطات:
- يدعم العربية الفصحى القانونية دعمًا حقيقيًا (لا مجرد multilingual سطحي).
- يعمل محليًا على الخادم (المصادر القانونية **لا تغادر الخادم** — راجع §6).
- بُعد ثابت مُوثَّق (768/1024 حسب النموذج).

**تبعة حتمية:** تغيير النموذج ⇒ تغيير `VECTOR_SIZE` ⇒ **collection جديدة**. لا يمكن خلط متجهات نموذجين. الخطة:

```text
legalmind_objects_v1  (384، hash، تجريبية)  →  تُحذف
legalmind_objects_v2  (بُعد النموذج، دلالية)  →  تُبنى من PostgreSQL
```

إعادة الفهرسة آمنة دائمًا لأن **PostgreSQL هو مصدر الحقيقة** و Qdrant فهرس مشتق ([[KNOWLEDGE_MODEL]] §2).

## 3. المسار المطلوب

```text
سؤال المستخدم

  → تحديد الفرع
  → تحديد الموضوع
  → تحديد عنوان تصنيف المحكمة   ← ضيّق هنا أولًا، لا بحث شامل ابتداءً
  → تحديد المسألة الدقيقة
  → تطبيق النطاق الزمني          ← استبعاد historical_only

```

## 4. الاسترجاع الهجين — القرار

البحث المتجهي وحده **لا يكفي** للنص القانوني. الاستشهاد الدقيق («المادة 338») مطلب لفظي حرفي لا دلالي.

```text
النتيجة = دمج(
    بحث متجهي   (Qdrant، للمعنى والمرادفات),
    بحث لفظي     (PostgreSQL FTS/BM25، للمواد والاستشهادات الحرفية),
    مُرشِّحات صارمة (branch, topic, subtopic, temporal, madhhab)
)
```

**المُرشِّحات ليست تحسينًا للترتيب — هي شرط صحة قانونية.** إرجاع مبدأ جعفري لسؤال سني ليس نتيجة ضعيفة، بل **جواب باطل**.

## 5. ترتيب الأسبقية

1. تطابق المسألة الدقيقة
2. تطابق عنوان تصنيف المحكمة
3. النص التشريعي النافذ
4. المبدأ القضائي المباشر
5. الحكم الكامل
6. القاعدة الجامعة المدعومة
7. الإجراء أو الدفع
8. النموذج القضائي

**قاعدة حاكمة:** نوع الطعن والدائرة **بيانات توثيقية لا تصنيفية**. المبدأ التجاري يُسترجع في الأحوال الشخصية إذا كان موضوعه كذلك — وهذا واقع: `JUR-PS-JURISDICTION-00020` طعن **تجاري** ويحكم **الإرث**.

## 6. قيد الخصوصية

المصادر القانونية وملفات القضايا **لا تغادر الخادم**. أي نموذج تضمين أو توليد يعمل **محليًا**، أو تُتخذ قرارات إرسال البيانات بموافقة صريحة من صاحب المشروع. هذا قيد افتراضي، لا خيار.

## 7. عقد الواجهة المقترح

```http
POST /api/search
{
  "query": "مسكن الحاضنة",
  "branch": "أحوال شخصية",
  "topic": null,
  "object_types": ["judicial_principle", "legislation"],
  "as_of": "2026-07-14",
  "madhhab": "sunni",
  "limit": 20
}
```

كل نتيجة **تجب** أن تحمل: `object_id`, `source_key`, `verification_status`, `temporal_scope`, `score`.
**نتيجة بلا سند = نتيجة مرفوضة.**

## 8. ما يجب أن تعرفه الواجهة

مربع البحث الحالي في `web/app/page.tsx` هو `Array.filter()` على بيانات مُحمَّلة في المتصفح. **يجب استبداله** بنداء `/api/search` عند بناء هذا المحرك — وإلا بقيت الواجهة تكذب على المستخدم.
