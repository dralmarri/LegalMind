## 1. المسار

```text
/opt/legalmind-ingest/inbox/
   file.{docx|pdf|html|txt|md}  +  file.<ext>.json  (sidecar)
        │
        ├── normalize()        ← طبقة التطبيع [[NORMALIZER]]
        │      أي صيغة → Canonical Markdown
        │      ما بعدها لا يعرف صيغة المصدر
        │
        ├── classify()         حسب source_type في الـ sidecar
        │      ├── legislation          → split_articles()   ^المادة N$
        │      ├── judicial_principle   → split_principles()  ^N-
        │      ├── full_judgment        → كائن واحد
        │      └── template/memorandum  → كائن واحد
        ├── content_digest()   ← SHA-256 على المتن المُطبَّع
        │      find_duplicate(branch, topic, digest) **قبل أي كتابة**
        │      تكرار؟ → دفعة `duplicate` + إظهار الدفعة السابقة + لا نسخة
        │
        ├── PostgreSQL: sources + knowledge_objects + ingestion_batches
        │      ← مصدر الحقيقة. يُكتب أولًا ويُلتزم أولًا.
        ├── Qdrant:     PUT points (E5 دلالي، 768 بُعدًا)
        │      ← فهرس مشتق. فشله لا يفقد معرفة: `reindex` يعيد بناءه.
        └── archive/    الأصل + *.canonical.md (أثر تدقيق)
             عند الخطأ → failed/ + وسم الدفعة `failed` مع سبب الرفض بالعربية
```

**ترتيب الكتابة ليس تفصيلًا:** PostgreSQL يُلتزم **قبل** الفهرسة. لو انعكس الترتيب لأمكن أن يوجد متجه لكائن لا وجود له — فهرس يكذب على مصدر الحقيقة.

## 2. عقد الـ Sidecar

لكل مصدر، ملف `.json` مجاور يكتبه `admin/app.py` — من `/api/upload` أو `/api/paste-text`:

```json
{
  "source_type": "judicial_principles_collection",
  "object_type": "judicial_principle",
  "branch": "أحوال شخصية",
  "topic": "الاختصاص",
  "subtopic": "اختصاص دائرة الأحوال الشخصية",
  "classification_title": "اختصاص دائرة الأحوال الشخصية",
  "micro_issue": "الدوائر السنية والجعفرية",
  "court_level": "تمييز",
  "circuit": "أحوال شخصية",
  "verification_status": "source_verified",
  "source_notes": "…",
  "upload_origin": "file_upload | pasted_text",
  "id_prefix": "JUR-PS-JURISDICTION",
  "source_key": "PS-JURISDICTION-SRC-0001",
  "title": "…"
}
```

**إلزامي:** `branch` + `topic` + `classification_title`. الواجهة ترفض دونها.

**`id_prefix` هو الحقل الأخطر.** بدونه يولّد المحرك المعرّف من `slug()` على اسم الملف — فتضيع الهوية الدائمة وتنكسر الاستشهادات. عند حقن المعرفة الموجودة، **يجب** تمريره صراحة.

**الـsidecar يُكتب بعد الملف**، فوجوده يعني أن المصدر مكتمل وجاهز للمعالجة — لا يلتقط المراقبُ ملفًا نصفَ مكتوب.

### لصق النص — `POST /api/paste-text`

| الشرط | التطبيق |
|---|---|
| طريقة واحدة لا اثنتان | الواجهة تعطّل الحقل غير المستخدم، فلا يصل ملف ونص معًا |
| رفض الفارغ أو < 20 حرفًا | `400` برسالة عربية |
| حد أعلى آمن | 500,000 حرف → `413` برسالة عربية |
| الأصل غير قابل للتغيير | يُحفظ Markdown حرفيًا كما لُصق، ثم `chmod 0444` |
| اسم ملف آمن فريد | `safe_stem()` يقبل العربية ويرفض المسارات والمحارف الخطرة، ويضيف لاحقة عشوائية عند التصادم |
| محرك الإدخال نفسه | لا مسار موازٍ — الملف يقع في `inbox` ويعالجه المحرك ذاته |
| `upload_origin=pasted_text` | يُحفظ في الـsidecar وفي تقرير الدفعة |
| إرجاع معرّف وحالة | `{"file_id": "…", "status": "queued"}` |

## 3. منع التكرار

**البصمة على المتن المُطبَّع، لا على بايتات الملف.** التطبيع التقني لا يمس المحتوى القانوني، فملف DOCX ونص ملصق يحملان النص نفسه يعطيان البصمة نفسها ويُعدّان تكرارًا — وهو المطلوب. لو بُصمت البايتات لأفلت التكرار بمجرد تغيير الصيغة.

- **النطاق:** `(branch, topic)`. النص نفسه تحت فرع أو موضوع مختلف إدخال مشروع.
- **القيد في قاعدة البيانات لا في التطبيق:** `idx_sources_content_dedup` فهرس فريد — لا يمكن الالتفاف عليه بكود جديد.
- **عند التكرار:** دفعة بحالة `duplicate` تحمل الدفعة السابقة، والملف يُؤرشف بلاحقة `__DUPLICATE__`. لا كائن ولا مصدر جديد.
- **idempotent:** إعادة إدخال النص نفسه لا تغيّر شيئًا ولا تفشل.

## 4. حالة التوثيق والسلطة

`verification_status` يأتي من المصدر (الـsidecar)، و**السلطة تُشتق منه ولا تُعلن** (`authority_for`):

| حالة التوثيق | السلطة | يُستشهد به؟ |
|---|---|---|
| `source_verified` | `source_authority` | ✅ |
| `operationally_accepted` | `human_verified_authority` | ✅ |
| `machine_pending_human` | `non_authoritative` | ❌ |
| `historical_only` وما بعدها | `non_authoritative` | ❌ |
| **`synthesized_rule`** (أي نوع) | `non_authoritative` | ❌ **أبدًا** |

القاعدة المستنبطة آليًا لا تصير سندًا بمجرد رفعها من الواجهة. تحرسها قيود قاعدة بيانات (`ko_synthesized_rule_never_citable`) لا عُرف تطبيقي يُنسى.

## 5. القيود المعروفة

| القيد | الأثر |
|---|---|
| ⚠️ لا OCR | PDF **النصي** مدعوم. **الممسوح ضوئيًا يُرفض** برسالة عربية مفهومة، وتُسجَّل دفعة `failed` تحمل السبب — بلا مصدر ولا كائن فارغ. |
| ❌ لا يكتب `relationships` | `relationship_count` يبقى 0 دائمًا |
| ❌ لا يكتب `verification_issues` | طابور التوثيق خارج قاعدة البيانات |
| ❌ لا يملأ `temporal_scope` | القيد الزمني ضائع رغم وجود العمود |
| ⚠️ الفرع الافتراضي `أحوال شخصية` | خطر صامت: ملف بلا `branch` يُصنَّف أحوالًا شخصية. الواجهة تفرض الحقل، لكن المحرك يقبل sidecar بلا فرع. |
| ⚠️ التقسيم بـ regex فقط | مادة بصيغة غير متوقعة تُبتلع في المادة السابقة |

## 6. الفشل مضمون الاحتواء

- **فشل التطبيع** (PDF ممسوح، صيغة تالفة): الملف → `failed/`، ودفعة `failed` تحمل **سبب الرفض بالعربية** ليقرأه المستخدم في الواجهة. **لا يبقى في `inbox`** — لو بقي لأعاد المراقب محاولته إلى ما لا نهاية.
- **فشل بعد بدء الكتابة:** الدفعة تُوسم `failed` صراحةً. دفعة عالقة على `started` كذبة إحصائية.
- **فشل الفهرسة:** لا يفقد معرفة. PostgreSQL التزم فعلًا، و`reindex` يعيد بناء الفهرس.

## 7. إعادة بناء الفهرس

Qdrant **فهرس مشتق يُرمى ويُعاد بناؤه**، لا مخزن معرفة:

```bash
cd /opt/LegalMind && set -a && . deploy/.env && set +a
PYTHONPATH=/opt/LegalMind .venv/bin/python engine/legalmind_engine.py reindex
```

يحذف الـcollection، يعيد إنشاءها، يعيد تضمين كل `knowledge_objects` من PostgreSQL دفعاتٍ من 64، ثم يتحقق من تطابق العدّين بالعدّ **الدقيق** (`points/count {"exact": true}`) — لا بالحقل المخزّن مؤقتًا `points_count` الذي يتأخر عن الحقيقة بعد الكتابة مباشرةً.
